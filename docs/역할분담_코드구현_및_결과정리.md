# FAA Wildlife Strike Damage Prediction 구현 및 결과 정리

작성 기준일: 2026-06-29  
목적: 팀 역할 분담을 위해 현재 저장소에 구현된 코드, 산출물, 모델 결과, 대시보드 기능, 검증 상태를 한 문서에서 확인할 수 있도록 정리한다.

---

## 1. 프로젝트 전체 구조

현재 프로젝트는 크게 두 부분으로 나뉜다.

1. `FAA_Wildlife_Damage_Final`
   - FAA wildlife strike 사고 데이터로 항공기 손상 여부(`Damage_Binary`)를 예측하는 모델링 파이프라인이다.
   - 데이터 전처리, train/test 분할, 내부 validation 분할, 모델 학습, threshold 선택, 최종 평가, 표/그래프/보고서/제출 패키지 생성을 담당한다.

2. `wildlife_strike_dashboard_vue`
   - 모델링 파이프라인에서 생성된 CSV 결과를 읽어 시각화하는 Vue 3 + TypeScript 대시보드이다.
   - 필터링, KPI, 손상률 차트, 비행 조건별 위험도, 야생동물 요인별 위험도, 빈도-심각도 매트릭스, 모델 성능 비교를 제공한다.

원본 데이터 파일은 저장소 루트의 `FAA_0619_preprocessed_dataset_for_team.xlsx`이며, 모델링 파이프라인은 이 파일의 `Final_Dataset` 시트를 읽는다.

## 2. 모델링 파이프라인 구현 내용

### 2.1 실행 진입점

파일: `FAA_Wildlife_Damage_Final/run_main_pipeline.py`

이 파일은 전체 모델링 작업의 메인 실행 파일이다. 주요 처리 순서는 다음과 같다.

1. 산출물 폴더 생성
2. 원본 Excel 파일 복사 및 로드
3. 데이터 누수 위험 컬럼 탐지
4. 중복 행 제거
5. Main Model 입력 변수 15개와 target만 남긴 모델링 데이터 생성
6. train/test stratified split
7. train 내부 fit/validation stratified split
8. validation 기준으로 모델 후보 비교
9. validation 기준으로 threshold 선택
10. train 전체 데이터로 최종 재학습
11. test 데이터에서 최종 평가
12. 예측 CSV, 성능 CSV, confusion matrix, 그래프, feature importance, 위험도 테이블 생성
13. 제출용 패키지와 zip 생성

### 2.2 핵심 공통 로직

파일: `FAA_Wildlife_Damage_Final/src/pipeline_core.py`

구현된 주요 상수와 함수는 다음과 같다.

`TARGET`
- 예측 대상 컬럼은 `Damage_Binary`이다.
- 0은 손상 없음, 1은 손상 있음이다.

`MAIN_FEATURES`
- 문서 기준 Main Model 입력 변수 15개만 사용한다.
- 사용 변수:
  - `INCIDENT_MONTH`
  - `TIME_OF_DAY`
  - `PHASE_OF_FLIGHT`
  - `ALTITUDE_GROUP`
  - `DISTANCE_GROUP`
  - `SPECIES`
  - `SIZE_CLEAN`
  - `NUM_STRUCK`
  - `AC_CLASS`
  - `AC_MASS`
  - `TYPE_ENG`
  - `NUM_ENGS`
  - `FAAREGION`
  - `STATE`
  - `SKY_CLEAN`

`detect_leakage_columns`
- 모델 입력에 들어가면 안 되는 데이터 누수 위험 컬럼을 탐지한다.
- 정확히 제외하는 컬럼 예:
  - `DAMAGE_LEVEL`
  - `INDICATED_DAMAGE`
  - `COST_REPAIRS`
  - `COST_OTHER`
  - `AOS`
  - `EFFECT`
  - `INDEX_NR`
  - `REG`
  - `FLT`
  - `REMARKS`
  - `COMMENTS`
  - `REMAINS_COLLECTED`
  - `REMAINS_SENT`
- prefix 기준으로 제외하는 계열:
  - `DAM_*`
  - `STR_*`
  - `ING_*`

`make_modeling_frame`
- 원본 데이터에서 `MAIN_FEATURES` 15개와 `Damage_Binary`만 남긴다.
- 누수 가능성이 있는 충돌 후 손상 기록 컬럼은 모델링 프레임에 포함하지 않는다.

`rare_category_maps`
- train 데이터 기준으로 희소 범주를 `Other`로 묶는다.
- 기본 기준은 `min_count=50`이다.
- validation/test 정보를 보고 범주 기준을 만들지 않도록 train 데이터만 사용한다.

`apply_category_maps`
- train에서 만든 희소 범주 mapping을 train/validation/test에 동일하게 적용한다.
- validation/test에서 처음 등장한 범주는 `Other`로 처리한다.

`stratified_splits`
- 손상 클래스 비율이 유지되도록 stratified split을 적용한다.
- 공식 train/test는 80:20으로 분할한다.
- train 내부에서 threshold 조정용 validation을 다시 분리한다.

`choose_threshold`
- validation 예측 확률만 사용해 최종 threshold를 선택한다.
- 후보 threshold는 0.05부터 0.95까지 0.05 간격과 기본 0.5이다.
- 선택 우선순위:
  - precision 40% 이상에서 recall 최대
  - precision 35% 이상에서 recall 최대
  - F2 최대
  - F1 최대
  - 기본 threshold 0.5

### 2.3 모델 후보

파일: `FAA_Wildlife_Damage_Final/run_main_pipeline.py`

구현된 모델은 3개이다.

1. Logistic Regression
   - `class_weight="balanced"`
   - `max_iter=1000`

2. Decision Tree
   - `max_depth=8`
   - `min_samples_leaf=20`
   - `class_weight="balanced"`

3. RandomForest
   - `n_estimators=200`
   - `max_depth=14`
   - `min_samples_leaf=10`
   - `class_weight="balanced"`
   - `n_jobs=-1`

클래스 불균형이 크기 때문에 세 모델 모두 balanced class weight를 사용한다.

### 2.4 전처리 방식

구현된 전처리 방식은 다음과 같다.

1. 결측값
   - 범주형 값은 mapping 단계에서 `Unknown`으로 처리된다.

2. 희소 범주 처리
   - train 기준 빈도 50 미만 범주는 `Other`로 묶는다.
   - 특히 `SPECIES`처럼 고유값이 많은 변수에서 과적합과 차원 폭증을 줄이기 위한 처리이다.

3. One-Hot Encoding
   - `OneHotEncoder(handle_unknown="ignore", sparse_output=False)`를 사용한다.
   - encoder는 train 또는 fit split에서만 학습하고, validation/test에는 transform만 적용한다.

4. 전처리 객체 저장
   - 최종 전처리 객체는 `FAA_Wildlife_Damage_Final/02_models/preprocessing_pipeline.pkl`로 저장된다.

- 이 모델은 최종 판정 도구라기보다 손상 가능성이 높은 사고를 우선 선별하는 screening 모델로 해석하는 것이 적절하다.

## 10. 현재 한계 및 주의사항

파일: `FAA_Wildlife_Damage_Final/07_reports/limitations.md`

현재 프로젝트의 해석상 한계는 다음과 같다.

1. 데이터는 보고된 wildlife strike 사고만 포함한다.
   - 모든 항공편이나 미보고 사고를 포함하지 않는다.

2. 모델은 strike 발생 가능성을 예측하지 않는다.
   - 이미 wildlife strike가 발생한 사고 중 항공기 손상 여부를 예측한다.

3. 공항별/항공사별 실제 위험도를 과도하게 주장하면 안 된다.
   - 노출량, 즉 전체 운항 횟수나 공항별 traffic denominator가 없다.

4. 희소 범주는 `Other`로 묶었다.
   - 과적합과 발표 안정성을 줄이는 장점이 있지만, 세부 종/세부 지역 해석은 일부 손실된다.

5. threshold 선택은 validation 데이터에서만 수행했다.
   - test set은 최종 평가에만 사용했다.

6. 최종 threshold 0.85는 precision을 높이는 선택이다.
   - 손상 의심 건을 더 엄격히 잡지만, 놓치는 손상 건도 늘어난다.
